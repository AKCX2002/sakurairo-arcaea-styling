第二章：开发环境搭建——从零开始配置 HPM SDK

这一章解决一个非常实际的问题：从一台全新的电脑开始，如何把 HPM SDK 的开发环境搭起来、跑通第一个工程。我假设你已经安装了 Git —— 如果还没装，先去 git-scm.com 下载安装，这是嵌入式开发的标配。

很多人会在这一步卡住——不是因为操作难，而是因为不知道安装顺序。正确的顺序是：

为什么是这个顺序？因为每一步的安装脚本或配置都会依赖上一步的结果。如果先装 VSCode 再装 Toolchain，VSCode 的 CMake Tools 插件在扫描编译器时就会找不到。下面按顺序逐一说明。

SDK 安装——git clone 与目录结构

HPM SDK 托管在 GitHub 上，使用 BSD-3 许可证。安装就是 clone 到本地：

# 克隆主仓库
git clone https://github.com/hpmicro/hpm_sdk.git

# 如果网络慢，可以使用 Gitee 镜像
git clone https://gitee.com/hpmicro/hpm_sdk.git

# 进入 SDK 目录
cd hpm_sdk

完成后，SDK 根目录的结构如下——每个文件夹都有明确的职责：

hpm_sdk/
  ├── arch/              RISC-V 架构层：CSR 寄存器读写、异常向量基址宏
  ├── soc/               SoC 层：与具体芯片型号绑定
  │   ├── HPM6800/       HPM6800 系列寄存器定义、start.S、system.c
  │   └── HPM6P00/       6P00 系列（高端型号）
  ├── boards/            板级支持：YAML 描述 + pinmux.c + board.c
  │   └── hpm6800evk/    示例：hpm6800evk.yaml → SoC=HPM6880, RAM=256MB
  ├── drivers/           外设驱动 API（多数为 header-only inline）
  │   ├── inc/           hpm_uart_drv.h, hpm_dmav2_drv.h, ...
  │   └── src/           驱动实现（可选）
  ├── middleware/        第三方中间件：FreeRTOS、lwIP、tinyUSB、vglite
  ├── components/        SDK 内置可复用组件
  ├── samples/           300+ 示例工程：hello_world、双核 Mailbox、LVGL…
  ├── cmake/             CMake 扩展函数：cmake-ext.cmake, application.cmake
  ├── scripts/           Python 构建脚本：build_linked_project.py
  └── utils/             辅助工具：crc32、swap、sbrk

设置环境变量指向 SDK 路径，每次打开新终端时都需要执行：

# Linux / macOS
export HPM_SDK_BASE=$HOME/hpm_sdk

# 也可以使用 SDK 自带的脚本
source $HPM_SDK_BASE/env.sh

# Windows (cmd)
set HPM_SDK_BASE=C:\hpm_sdk
# 或执行 env.cmd
建议把 source env.sh 加到 .bashrc 或 .zshrc 中，省得每次开终端都要手动 export。

Toolchain 安装——xPack GCC 与官方 GCC

HPM SDK 默认使用 GNU GCC for RISC-V。有两种获取方式，任选其一即可。

方式一：HPMicro 官方 GCC（推荐）

从 HPMicro 官方 GitHub Releases 页面下载预编译的 RISC-V 工具链：

# 下载地址（根据系统选择对应版本）
# https://github.com/hpmicro/riscv-gnu-toolchain/releases

# Linux x86_64 示例
wget https://github.com/hpmicro/riscv-gnu-toolchain/releases/download/2024.08/riscv32-unknown-elf.gcc-13.2.0.tar.gz
tar xzf riscv32-unknown-elf.gcc-13.2.0.tar.gz -C /opt/

# 设置环境变量
export GNURISCV_TOOLCHAIN_PATH=/opt/riscv32-unknown-elf.gcc-13.2.0
export HPM_SDK_TOOLCHAIN_VARIANT=       # 空值 = gnu-gcc
export PATH=$GNURISCV_TOOLCHAIN_PATH/bin:$PATH

# 验证
riscv32-unknown-elf-gcc --version

方式二：xPack RISC-V GCC

xPack 提供了一种跨平台的包管理方式，特别适合 macOS 和 Windows：

# 使用 npm 安装
npm install --global @xpack-dev-tools/riscv-none-elf-gcc@latest

# 或者使用 xpm
xpm install --global @xpack-dev-tools/riscv-none-elf-gcc@latest

# 安装后设置环境变量
export GNURISCV_TOOLCHAIN_PATH=$HOME/.local/xpack-riscv-none-elf-gcc-13.2.0-2
export HPM_SDK_TOOLCHAIN_VARIANT=

# Windows 示例
set GNURISCV_TOOLCHAIN_PATH=C:\Users\username\AppData\Roaming\xPacks\riscv-none-elf-gcc\13.2.0-2
set HPM_SDK_TOOLCHAIN_VARIANT=
无论用哪种方式，安装完成后务必验证 riscv32-unknown-elf-gcc --version 或 riscv-none-elf-gcc --version 能正确输出版本号。最常见的搭建失败原因就是环境变量没设对——CMake 在 Configure 阶段会检查 GNURISCV_TOOLCHAIN_PATH 指向的 bin/ 目录下是否存在可执行文件。

CMake 安装

HPM SDK 要求 CMake 版本不低于 3.13。三条命令覆盖三大平台：

# Linux (Ubuntu/Debian)
sudo apt install cmake

# macOS (Homebrew)
brew install cmake

# Windows (Chocolatey, 需要管理员权限)
choco install cmake --installargs 'ADD_CMAKE_TO_PATH=System'

验证安装：

cmake --version
# 应该输出 cmake version 3.13.0 或更高

如果你用的 Linux 发行版自带的 CMake 版本过旧（比如 Ubuntu 18.04 只有 3.10），可以从 Kitware 官方仓库安装最新版：

# 添加 Kitware 签名密钥
wget -O - https://apt.kitware.com/keys/kitware-archive-latest.asc | gpg --dearmor | sudo tee /usr/share/keyrings/kitware-archive-keyring.gpg

# 添加仓库（以 Ubuntu 22.04 为例）
echo "deb [signed-by=/usr/share/keyrings/kitware-archive-keyring.gpg] https://apt.kitware.com/ubuntu/ $(lsb_release -sc) main" | sudo tee /etc/apt/sources.list.d/kitware.list

sudo apt update && sudo apt install cmake

Ninja 安装——为什么不用 Make？

很多开发者第一次接触 HPM SDK 时都会发现：官方推荐用 Ninja 而不是传统的 Make。这不是为了赶时髦，而是有实际理由的。

Ninja vs Make：核心区别

Make 是一个通用的构建工具，它自己就能解析 Makefile、处理规则、管理依赖。Ninja 不同——它只做一件事：按照构建描述文件（build.ninja）执行编译命令，不解析任何构建逻辑。

这意味着 Ninja 本身没有条件判断、没有循环、没有变量展开。所有逻辑都在 CMake 的 Configure 阶段完成，生成的 build.ninja 是一个纯粹的编译命令序列。Ninja 要做的只是尽可能快地执行这些命令。

对比维度          Make                  Ninja
──────────────────────────────────────────────────
解析规则语言      自行解析 Makefile        不解析，读 build.ninja
并行编译          需要 -j 参数指定       默认全核并行
重启编译速度      中等（重新解析 Makefile）  极快（纯命令队列）
增量编译精确度    基于文件时间戳            基于 depfile + 时间戳
内存占用          中等                    极低
Windows 支持      需要 MinGW              原生支持

嵌入式项目的特点决定了 Ninja 的优势：一个典型的 HPM SDK 工程包含数百个源文件（SDK 自身 + middleware + 应用代码），每次修改一行代码后，Make 重新解析 Makefile 需要 1~2 秒，而 Ninja 可以在 50 毫秒内决定哪些文件需要重新编译。对于"改一行 printf 等 10 秒编译"的调试循环，Ninja 能省掉大量等待时间。

安装命令

# Linux
sudo apt install ninja-build

# macOS
brew install ninja

# Windows
choco install ninja

# 验证
ninja --version

VSCode 推荐插件

虽然 HPM SDK 是命令行优先的构建系统，但 VSCode 作为编辑器仍然是最好的选择。以下三个插件是必备的：

1. CMake Tools（ms-vscode.cmake-tools）

这是 CMake 生态中最核心的 VSCode 插件。它提供：

  • CMakeLists.txt 语法高亮和智能补全
  • Configure / Build / Debug 按钮（替代命令行操作）
  • Kit 选择器：配置 Toolchain 和生成器（Ninja）
  • Target 列表：直接选择编译哪个子工程

安装后在 .vscode/settings.json 中配置：

{
    "cmake.configureSettings": {
        "BOARD": "hpm6800evk",
        "CMAKE_MAKE_PROGRAM": "/usr/bin/ninja"
    },
    "cmake.generator": "Ninja",
    "cmake.buildDirectory": "${workspaceFolder}/build"
}

2. Cortex-Debug（marus25.cortex-debug）

这是嵌入式 GDB 调试的核心插件。配合 OpenOCD 或 JLink GDB Server，可以在 VSCode 里完成断点调试、寄存器查看、外设寄存器浏览。需要在 .vscode/launch.json 中配置：

{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "HPM OpenOCD Debug",
            "type": "cortex-debug",
            "request": "launch",
            "servertype": "openocd",
            "cwd": "${workspaceFolder}",
            "executable": "build/output/demo.elf",
            "device": "HPM6880",
            "configFiles": [
                "${env:HPM_SDK_BASE}/boards/openocd/probes/ft2232.cfg",
                "${env:HPM_SDK_BASE}/boards/openocd/soc/hpm6800-single-core.cfg",
                "${env:HPM_SDK_BASE}/boards/openocd/boards/hpm6800evk.cfg"
            ],
            "svdFile": "${env:HPM_SDK_BASE}/soc/HPM6800/HPM6880/hpm6880.svd"
        }
    ]
}

3. clangd（llvm-vs-code-extensions.vscode-clangd）

clangd 提供基于 LLVM 的 C/C++ 智能感知，比微软官方的 C/C++ IntelliSense 更快、更准确。它会读取 compile_commands.json（由 CMake 生成），精确知道每个文件的宏定义和 include 路径。配置如下：

{
    "clangd.arguments": [
        "--background-index",
        "--compile-commands-dir=${workspaceFolder}/build",
        "--query-driver=/opt/riscv32-unknown-elf.gcc-13.2.0/bin/riscv32-unknown-elf-gcc"
    ]
}
注意：clangd 需要 CMake 在 Configure 阶段生成 compile_commands.json。如果 build 目录下没有这个文件，可以在 CMakeLists.txt 中加一行 set(CMAKE_EXPORT_COMPILE_COMMANDS ON) 或者在命令行加 -DCMAKE_EXPORT_COMPILE_COMMANDS=ON。

OpenOCD 安装——三种调试器配置

OpenOCD 负责将 GDB 的调试命令翻译为目标芯片的 JTAG/SWD 协议。HPMicro 维护了一个专门的分支，添加了 Flash 烧录算法支持。

# 克隆 HPMicro 定制的 OpenOCD
git clone https://github.com/hpmicro/riscv-openocd.git --depth=1

# 编译（需要 autoconf, automake, libtool, libusb-1.0）
cd riscv-openocd
./bootstrap
./configure --enable-ftdi --enable-jlink
make -j$(nproc)
sudo make install

也可以直接下载预编译的二进制包（如果官方 Releases 提供了）：

# 下载后解压，把 openocd 可执行文件加入 PATH 即可
export PATH=/path/to/openocd/bin:$PATH
openocd --version

HPM SDK 在三类调试器都做了完整支持，每种在 boards/openocd/probes/ 下都有对应的配置文件：

CMSIS-DAP（调试器配置文件：probes/cmsis_dap.cfg）

CMSIS-DAP 是最常用的低成本调试器，HPMicro 官方评估板集成了板载 CMSIS-DAP。启动命令：

openocd \
    -f probes/cmsis_dap.cfg \
    -f soc/hpm6800-single-core.cfg \
    -f boards/hpm6800evk.cfg

FT2232（调试器配置文件：probes/ft2232.cfg）

FT2232 是 FTDI 系列的高性能调试器，支持最高 30MHz 的 JTAG 时钟。常用于第三方调试器如 J-Link OB 或自制调试器：

openocd \
    -f probes/ft2232.cfg \
    -f soc/hpm6800-single-core.cfg \
    -f boards/hpm6800evk.cfg

J-Link（调试器配置文件：probes/jlink.cfg）

Segger J-Link 是业界标准调试器，支持最高 50MHz JTAG 时钟和 Unlimited Flash Breakpoints。注意 J-Link 需要安装 Segger 的 J-Link 驱动：

# 下载 J-Link Software Pack: https://www.segger.com/downloads/jlink/
# 安装后 J-Link GDB Server 可用
# OpenOCD 端配置
openocd \
    -f probes/jlink.cfg \
    -f soc/hpm6800-single-core.cfg \
    -f boards/hpm6800evk.cfg
Linux 用户注意事项：使用 FT2232 或 CMSIS-DAP 时，需要安装 udev 规则才能以非 root 用户访问 USB 设备。SDK 已提供这些规则文件：sudo cp $HPM_SDK_BASE/scripts/linux/udev_rules/*.rules /etc/udev/rules.d/ && sudo udevadm control --reload-rules。

验证：编译 hello_world

安装完成后，用 hello_world 示例验证整个工具链是否通畅：

cd $HPM_SDK_BASE/samples/hello_world
cmake -B build -GNinja -DBOARD=hpm6800evk
cmake --build build

# 确认输出文件
ls build/output/
# 应该看到: demo.elf demo.bin demo.hex demo.map demo.asm

如果输出目录中看到了 demo.elf 和 demo.bin，说明开发环境已经完全搭建成功。

第三章：第一个工程——cmake -B build 到底发生了什么

绝大多数 HPM SDK 教程只会教你跑两条命令：

cd samples/hello_world
cmake -B build -GNinja -DBOARD=hpm6800evk
cmake --build build

然后告诉你"编译成功了"。但对于一个想真正掌握这套工具的开发者来说，最大的困惑往往不是这两条命令怎么用，而是 为什么敲两条命令就能生成固件？中间发生了什么？出了问题该查哪里？

这一章会逐行分析 hello_world 工程的 CMakeLists.txt，然后分别拆解 Configure 阶段和 Build 阶段到底发生了什么。读完这一章，你应该能回答：SDK 怎么找到的？Board 怎么选中的？编译器怎么确定的？链接脚本怎么生效的？以及最关键的问题：改一行配置后，哪些文件会被重新编译？

Hello World 工程的 CMakeLists.txt

先看完整的工程文件。这是 SDK 中 samples/hello_world/ 目录的真实内容：

# samples/hello_world/CMakeLists.txt
# Copyright (c) 2021 HPMicro
# SPDX-License-Identifier: BSD-3-Clause

cmake_minimum_required(VERSION 3.13)

find_package(hpm-sdk REQUIRED HINTS $ENV{HPM_SDK_BASE})

project(hello_world)

sdk_app_src(src/hello_world.c)
generate_ide_projects()

只有 11 行，去掉注释和版权头其实只有 5 条有效语句。但这 5 条语句背后涉及了整个 SDK 构建系统的运作机制。

// samples/hello_world/src/hello_world.c
#include 
#include "board.h"
#include "hpm_debug_console.h"

#define LED_FLASH_PERIOD_IN_MS 300

int main(void)
{
    int u;
    board_init();
    board_init_led_pins();
    board_timer_create(LED_FLASH_PERIOD_IN_MS, board_led_toggle);

    printf("hello world\n");
    while (1) {
        u = getchar();
        if (u == '\r') {
            u = '\n';
        }
        printf("%c", u);
    }
    return 0;
}

这个 main.c 做的事情非常简单：初始化板子、点灯、打印 "hello world"、回显串口输入。但它用到了三个关键 API：board_init()（来自 board.c）、board_init_led_pins()（来自 pinmux.c）、printf（来自 newlib 库函数）。这些函数的源代码散布在 SDK 的 boards/、drivers/ 和 middleware/ 目录中。CMake 的 Configure 阶段要做的就是——找到所有这些依赖，准确地传递给编译器。

cmake -B build——Configure 阶段详解

输入 cmake -B build -GNinja -DBOARD=hpm6800evk 后，CMake 会执行一个叫做 Configure 的过程。这个过程不会编译任何代码，它的输出是 build/build.ninja——一份完整的编译命令清单。

让我们逐层拆解 Configure 阶段内部发生了什么：

sequenceDiagram
    participant User as cmake -B build
    participant CML as CMakeLists.txt
    participant Find as find_package(hpm-sdk)
    participant Config as hpm-sdk-config.cmake
    participant App as application.cmake
    participant Board as Board YAML + Toolchain
    participant Ext as cmake-ext.cmake
    participant SDK as hpm_sdk/CMakeLists.txt
    participant Ninja as build.ninja

    User->>CML: 解析 CMakeLists.txt
    CML->>Find: cmake_minimum_required + project()
    Find->>Config: 搜索 HPM_SDK_BASE 路径
    Config->>App: include(application.cmake)
    App->>Board: 解析 BOARD=hpm6800evk.yaml
    Board-->>App: SoC=HPM6880, RAM=256MB, Flash=32MB
    App->>Board: 检查 GNURISCV_TOOLCHAIN_PATH
    Board-->>App: 找到 riscv32-unknown-elf-gcc
    App->>Ext: include(cmake-ext.cmake)
    Ext-->>App: 注册 sdk_src/sdk_inc/sdk_compile_definitions
    App->>SDK: add_subdirectory(${HPM_SDK_BASE})
    SDK-->>App: 编译 HPM_SDK_LIB 的所有源文件清单
    App-->>CML: sdk_app_src → app OBJECT target
    CML->>Ninja: generate_ide_projects → build.ninja
    Ninja-->>User: Configure 完成，可执行 cmake --build

第一步：find_package(hpm-sdk REQUIRED)

CMake 的 find_package() 会在多个预设路径中搜索名为 hpm-sdk-config.cmake 的文件。由于我们在命令行中传了 HINTS $ENV{HPM_SDK_BASE}，它会优先在 $HPM_SDK_BASE 指向的目录下搜索。找到这个文件后，CMake 执行里面的配置。

hpm-sdk-config.cmake 的核心工作是设置几个全局变量并把控制权交给 application.cmake：

# hpm-sdk-config.cmake（简化版）
set(HPM_SDK_BASE "${CMAKE_CURRENT_LIST_DIR}" CACHE PATH "SDK base directory")
include(${HPM_SDK_BASE}/cmake/application.cmake)

第二步：application.cmake——构建系统的骨架

application.cmake（位于 cmake/application.cmake，355 行）是整个构建系统的核心。它做以下几件事：

创建三个 CMake Target：

# INTERFACE 库——传播头文件路径、编译选项、链接选项
add_library(hpm_sdk_lib_itf INTERFACE)

# STATIC 库——承载 SDK 所有 .c/.S 编译产物
add_library(hpm_sdk_lib STATIC "")

# OBJECT 库——承载应用代码
add_library(app OBJECT "")

# Executable——最终的链接产物
add_executable(demo.elf $<TARGET_OBJECTS:app>)

这里的设计非常干净：

  • hpm_sdk_lib_itf（INTERFACE）只管传播——编译选项、头文件路径、链接选项通过这些传给所有依赖方
  • hpm_sdk_lib（STATIC）承载 SDK 自身所有源文件的编译产物
  • app（OBJECT）承载你的应用代码
  • demo.elf 把 app 的 object 和 hpm_sdk_lib 链接在一起

检查环境变量：

if(NOT DEFINED ENV{GNURISCV_TOOLCHAIN_PATH})
    message(FATAL_ERROR "GNURISCV_TOOLCHAIN_PATH is not set yet")
endif()

这也是为什么前面强调必须先设置 Toolchain 环境变量的原因——Configure 阶段就会报错，不会等到 Build 阶段才崩溃。

解析 Board YAML：

# 在 SDK 的 boards/ 目录下搜索 BOARD 对应的 yaml 文件
find_path(SDK_BOARD_DIR NAMES ${BOARD}.yaml
    PATHS ${HPM_SDK_BASE}/boards/* NO_DEFAULT_PATH)

set(BOARD_YAML ${HPM_BOARD_DIR}/${BOARD}.yaml)
if(NOT EXISTS ${BOARD_YAML})
    message(FATAL_ERROR "No board named '${BOARD}' found")
endif()

hpm6800evk.yaml 的内容类似：

board:
    name: hpm6800evk
    soc: HPM6880
    device: HPM6880x1
    flash: 33554432        # 32MB
    extram: 268435456      # 256MB DDR

从 YAML 中提取出 SoC 型号、Flash 和外部 RAM 大小后，CMake 就知道该用哪个 SoC 的 start.S、哪个 Linker Script、以及支持哪些构建类型（ram / flash_xip / flash_sdram_xip 等）。

加载 cmake-ext.cmake 并编译 SDK：

include(${HPM_SDK_BASE}/cmake/cmake-ext.cmake)
enable_language(C CXX ASM)
add_subdirectory(${HPM_SDK_BASE} ${__build_dir})

add_subdirectory(${HPM_SDK_BASE}) 会递归执行 SDK 根目录的 CMakeLists.txt，其中通过 sdk_src() 函数收集了数百个驱动源文件和 middleware 源文件，全部添加到 hpm_sdk_lib 目标中。

第三步：sdk_app_src——你的应用代码

sdk_app_src(src/hello_world.c) 这个 SDK 自定义函数的底层调用是：

# cmake-ext.cmake 中的定义
function(sdk_app_src)
    target_sources(app PRIVATE ${ARGV})
endfunction()

把源文件添加到 app（OBJECT 库）中。OBJECT 库的好处是：源文件会被编译为 .o 文件，但不会打包成静态库，最终直接链接到 demo.elf。这避免了静态库链接时默认的"未引用符号不链接"行为。

第四步：generate_ide_projects——输出构建文件

最后 generate_ide_projects() 的作用取决于当前的 CMake Generator：

  • 使用 -GNinja：生成 build/build.ninja + build/compile_commands.json
  • 使用 -G"Unix Makefiles"：生成 build/Makefile
  • 如果是 Segger Embedded Studio：同时生成 .emProject 文件
  • 如果是 IAR：生成 .ewp 文件

这时 Generate 阶段完成，build/ 目录下已经有了完整的构建描述，但还没有一个 .o 文件。

cmake --build build——Build 阶段详解

cmake --build build 等价于在 build/ 目录下执行 ninja。Ninja 读取 build.ninja，按照依赖关系依次执行编译命令。

步骤 1：依赖分析

Ninja 读取 build/.ninja_deps 和 build/.ninja_log，比较每个 .o 文件的修改时间与其对应 .c/.S 文件以及所有被 #include 的头文件的修改时间（depfile 机制）。只有时间戳晚于输出的输入文件才会被重新编译。这就是"增量编译"的原理。

步骤 2：编译（Compilation）

Ninja 并行调用 GCC。每个源文件的编译命令类似：

riscv32-unknown-elf-gcc \
    -mcpu=andalus \
    -march=rv32imafdc \
    -mabi=ilp32f \
    -Os -g3 \
    -I$HPM_SDK_BASE/boards/hpm6800evk \
    -I$HPM_SDK_BASE/soc/HPM6800/HPM6880 \
    -I$HPM_SDK_BASE/drivers/inc \
    -I$HPM_SDK_BASE/arch/riscv \
    -c src/hello_world.c \
    -o build/build_tmp/CMakeFiles/app.dir/src/hello_world.c.o

注意命令行中数十个 -I 参数——它们是在 Configure 阶段由 sdk_inc() 递归收集并存储到 INTERFACE 目标中的。这正是 Configure 阶段和 Build 阶段的分工：Configure 阶段负责搜索和收集，Build 阶段负责执行和执行。

步骤 3：链接（Linking）

所有 .o 文件编译完成后，Ninja 调用链接器：

riscv32-unknown-elf-gcc \
    -mcpu=andalus \
    -march=rv32imafdc \
    -mabi=ilp32f \
    -T $HPM_SDK_BASE/soc/HPM6800/HPM6880/toolchains/gcc/ram.ld \
    -Wl,--gc-sections \
    -Wl,-Map=build/output/demo.map \
    -o build/output/demo.elf \
    build/build_tmp/CMakeFiles/app.dir/src/hello_world.c.o \
    build/build_tmp/lib/hpm_sdk_lib.a \
    -lc -lnosys

关键点：

  • -T ram.ld：使用 SoC 对应的链接脚本。Linker Script 定义了内存布局——ILM、DLM、SRAM、DDR 各自的基址和大小
  • -Wl,--gc-sections：垃圾回收未使用的函数和数据段，减少固件体积
  • -Wl,-Map=demo.map：生成内存映射文件，用于排查 RAM/Flash 占用
  • hpm_sdk_lib.a：SDK 静态库，包含所有驱动的编译产物
  • -lc -lnosys：链接 newlib C 库和最小化 syscall 实现

步骤 4：后处理（Post-processing）

application.cmake 自动添加了两个自定义命令到 demo.elf 目标：

# 从 ELF 生成二进制文件（用于烧录）
add_custom_command(TARGET demo.elf
    COMMAND ${CMAKE_OBJCOPY} -O binary -S
        ${EXECUTABLE_OUTPUT_PATH}/demo.elf
        ${EXECUTABLE_OUTPUT_PATH}/demo.bin)

# 从 ELF 生成汇编清单（用于代码审查）
add_custom_command(TARGET demo.elf
    COMMAND ${CMAKE_OBJDUMP} -S -d
        ${EXECUTABLE_OUTPUT_PATH}/demo.elf
        > ${EXECUTABLE_OUTPUT_PATH}/demo.asm)

因此编译完成后，build/output/ 目录下会包含：

build/output/
  ├── demo.elf        # ELF 格式，包含调试信息
  ├── demo.bin        # 纯二进制，用于烧录到 Flash
  ├── demo.hex        # Intel HEX 格式，部分烧录工具需要
  ├── demo.map        # 内存映射，查看函数/变量的地址分配
  └── demo.asm        # 反汇编清单，用于代码审查和性能分析

常见错误及排查方向

了解构建流程的最大好处是：问题出在哪个阶段，你立刻能定位。下面是三个最常见的错误及其根因：

错误 1：找不到 SDK（Configure 阶段）

CMake Error at CMakeLists.txt:5 (find_package):
  By not providing "Findhpm-sdk.cmake" in CMAKE_MODULE_PATH this project
  has asked CMake to find a package configuration file provided by
  "hpm-sdk", but CMake did not find one.

根因：HPM_SDK_BASE 环境变量未设置或指向了错误的目录。

排查方向：

  • 执行 echo $HPM_SDK_BASE（Linux/macOS）或 echo %HPM_SDK_BASE%（Windows）确认变量已设置
  • 确认 $HPM_SDK_BASE 目录下存在 cmake/hpm-sdk-config.cmake 文件
  • 通配路径：find_package(hpm-sdk REQUIRED HINTS $ENV{HPM_SDK_BASE}) 中的 HINTS 路径要准确
  • 如果使用 source env.sh，确认执行成功且没有错误输出

一句话定位：在工程目录下执行 cmake -B build -GNinja -DBOARD=hpm6800evk -DHPM_SDK_BASE=$HPM_SDK_BASE 2>&1 | head -20，前几行会明确告诉你 CMake 在哪些路径搜索了 hpm-sdk-config.cmake。

错误 2：找不到 Toolchain（Configure 阶段）

CMake Error at cmake/application.cmake:8 (message):
  GNURISCV_TOOLCHAIN_PATH is not set yet

根因：GNURISCV_TOOLCHAIN_PATH 环境变量未设置。

排查方向：

  • 确认 echo $GNURISCV_TOOLCHAIN_PATH 输出了正确路径
  • 确认该路径下的 bin/ 目录中存在 riscv32-unknown-elf-gcc（或 riscv32-elf-gcc，取决于工具链变体）
  • 如果设置了 HPM_SDK_TOOLCHAIN_VARIANT=nds-gcc，则搜索的是 riscv32-elf-gcc
  • 检查 PATH 是否包含 toolchain 的 bin 目录：export PATH=$GNURISCV_TOOLCHAIN_PATH/bin:$PATH
一个极隐蔽的坑：如果同时安装了 WSL 的 Windows 工具链和 WSL 内部的 Linux 工具链，CMake 可能找到错误架构的编译器。确认你设置的 GNURISCV_TOOLCHAIN_PATH 指向的是 Linux x86_64 版本的 Toolchain。

错误 3：找不到 Board（Configure 阶段）

CMake Error at cmake/application.cmake:73 (message):
  No board named 'hpm6800evk' found

根因：-DBOARD= 参数指定的板子名称在 SDK 中不存在。

排查方向：

  • 列出 SDK 中所有可用的板子：ls $HPM_SDK_BASE/boards/
  • 确认每个板子目录下有对应的 ${BOARD}.yaml 文件，例如 hpm6800evk/hpm6800evk.yaml
  • 常见的拼写错误：大小写、漏了 'evk' 后缀、把 '0' 写成了 'O'

错误 4：链接错误——undefined reference（Build 阶段）

riscv32-unknown-elf-gcc: error: build/build_tmp/lib/hpm_sdk_lib.a: No such file or directory
# 或者
undefined reference to `board_init'

根因：链接器在 CMake 的 target 依赖链中找不到对应的符号。

排查方向：

  • 确认 sdk_app_src() 包含了所有需要的源文件
  • 如果使用了 middleware（如 FreeRTOS），确认 app.yaml 中声明了对应的依赖，或者 CMakeLists.txt 中有 sdk_inc() 和 sdk_src() 调用
  • 检查 build/output/demo.map 文件，搜索缺失的符号名，看它是否被 GC 优化掉了（--gc-sections 会删除未调用的函数）
  • 如果定义了 weak 符号（如 __attribute__((weak))），确保强符号定义存在于某个源文件中

一个实用技巧：在 CMakeLists.txt 中加入 set(CMAKE_VERBOSE_MAKEFILE ON)，然后重新 cmake --build build，可以看到完整的编译和链接命令。从命令中能直接看出哪些 .o 参与了链接，哪些库被传给了链接器。这是排查链接问题的最有力手段。

总结

两条命令，两个完全不同的阶段。Configure 阶段是"问诊把脉"——检查环境、解析配置、生成计划。Build 阶段是"动手治疗"——执行编译、完成链接、输出文件。

理解了这个分界线，你就能做到：

  • 看到 find_package 错误 → 立刻检查 HPM_SDK_BASE
  • 看到 GNURISCV_TOOLCHAIN_PATH 错误 → 立刻检查 Toolchain 路径
  • 看到 undefined reference → 检查源文件列表和依赖声明
  • 看到编译成功但调试器无法连接 → 检查 Linker Script 和烧录配置
